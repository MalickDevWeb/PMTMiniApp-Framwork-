1. Vue d’ensemble

Présentation du framework

Ce qu’est PMTMiniApp

`PMTMiniApp` est un framework frontend pour WeChat Mini Program. Il ne remplace pas ton backend : il structure le projet, centralise la configuration, les services, l’auth, l’API, les capacités appareil et les conventions de page.

Ce qu’il n’est pas

Ce n’est pas un backend, ce n’est pas un ORM, et ce n’est pas un moteur de sécurité serveur. Le backend reste la source de vérité pour les données, les autorisations et les sessions réelles.

Ce que le développeur modifie vraiment

En pratique, le développeur d’application touche surtout `config/`, `pages/`, `routage/routes.json`, les composants métier et quelques services applicatifs. Le cœur `core/` reste stable dans la majorité des projets.

Règle centrale

Une page ne doit pas porter seule toute la logique du projet. Elle affiche, lit les services du framework, lance des actions claires, puis laisse `core/`, `config/`, `authService`, `apiService`, `configurationAppareil` et `gestionnaireEtat` faire le travail.

Si tu veux brancher un backend

Commence par `config/environments.js`, puis ajuste `config/authProvider.js` si le sujet touche l’authentification.

Si tu veux créer un écran

Utilise `create:page`, puis travaille dans `pages/`, pas directement dans `app.json` ou le cœur du framework.

Si tu veux un service natif

Passe par `serviceApp().configurationAppareil` avant d’aller fouiller dans les sous-services avancés.

Si tu veux valider un formulaire

Utilise `creerFormHelper` et `utils/validators` pour éviter de réécrire la logique de saisie et de messages.

2. Installation

Démarrage rapide

Créer un projet

npx create-pmtminiapp mon-app

Ou :

npm create pmtminiapp mon-app

Mettre à jour un projet existant

npx create-pmtminiapp update mon-app

Ou depuis la racine du projet :

npm run framework:update

Vérification complète du starter

npm run framework:ready

Cette commande enchaîne :

  • `sync:pages`
  • `doctor:navigation`
  • tous les tests du framework

Scripts de base à connaître

npm run create:page -- nomPage
npm run remove:page -- nomPage
npm run repair:pages
npm run sync:pages
npm run doctor
npm run check
Important

`routes.json` décrit la navigation. Il ne crée pas les fichiers physiques d’une page. Pour créer une vraie page, utilise `npm run create:page -- nomPage`.

La source de vérité du routage via `routes.json`.

5. Configuration

Les fichiers qu’un développeur modifie vraiment

`config/environments.js`

Définit l’environnement actif, `baseUrl`, `prefix`, `timeout`, headers, logs, flags et stratégie auth.

const envConfig = Object.freeze({
  currentEnv: 'prod',
  environments: Object.freeze({
    prod: Object.freeze({
      api: Object.freeze({
        baseUrl: 'https://api.exemple.com',
        prefix: '/v1',
        timeoutMs: 10000,
        defaultHeaders: Object.freeze({}),
        retry: Object.freeze({
          enabled: true,
          count: 2,
          delayMs: 300,
          backoffFactor: 2,
          retriableStatusCodes: Object.freeze([408, 429, 500, 502, 503, 504]),
        }),
        mockEnabled: false,
      }),
      auth: Object.freeze({
        strategy: 'api',
        redirectAfterLogin: 'index',
        redirectIfGuest: 'login',
      }),
      logs: Object.freeze({
        enabled: false,
        api: false,
      }),
      featureFlags: Object.freeze({}),
    }),
  }),
})

`config/authProvider.js`

Décrit comment ton backend auth fonctionne : transport, champs login, endpoints, rôles, OTP, social, refresh et mapping avancé.

`config/configuration_Appareil.js`

Active et règle les capacités natives : appareil, IP publique, permissions, Bluetooth, notifications et stratégie d’accès.

Bon réflexe

Commence toujours par `config/`. Si tu te retrouves à modifier `core/` pour brancher un backend classique, c’est souvent que la configuration n’a pas été assez exploitée.

6. Authentification

Brancher n’importe quel backend auth proprement

L’objectif du framework est simple : la page `login` reste stable, et le backend est branché surtout via `config/environments.js` et `config/authProvider.js`.

Cas simple : backend Bearer classique

const authProvider = Object.freeze({
  transport: 'bearer',

  loginFields: Object.freeze({
    identifiant: 'email',
    motDePasse: 'password',
  }),

  endpoints: Object.freeze({
    login: '/auth/login',
    me: '/auth/me',
    logout: '/auth/logout',
  }),

  redirectsByRole: Object.freeze({
    ADMIN: 'dashboardAdmin',
    USER: 'index',
  }),
})

Login social via backend

Le framework expose `authService.loginSocial(...)`. La sécurité reste côté backend. Le frontend consomme seulement le contrat backend, pas la logique OAuth sensible. 

endpoints: Object.freeze({
  login: '/auth/login',
  me: '/auth/me',
  social: '/auth/social',
  logout: '/auth/logout',
}),

socialProviders: Object.freeze({
  google: Object.freeze({
    enabled: true,
    mode: 'code',
  }),
}),

social: Object.freeze({
  providerField: 'provider',
  codeField: 'code',
  tokenField: 'token',
  idTokenField: 'idToken',
  redirectUriField: 'redirectUri',
})

Cas Google avec redirection backend

Important

Le framework ne pilote pas automatiquement l’ouverture navigateur Google dans le core. Le pattern recommandé est : le backend gère la redirection OAuth Google, puis le frontend récupère le `code` ou laisse le backend créer la session, puis appelle `authService.loginSocial(...)` ou `authService.restaurer()`. 

const { authService } = serviceApp()

const resultat = await authService.loginSocial({
  provider: 'google',
  code: codeGoogle,
  redirectUri: 'https://ton-retour-oauth',
})

7. Couche HTTP

API, verbes HTTP et organisation du code réseau

Dans PMTMiniApp, les pages ne doivent presque jamais appeler `wx.request` directement. Elles passent par `apiService`, puis idéalement par un client métier dédié.

Méthodes disponibles

const { apiService } = serviceApp()

await apiService.get('/clients', { auth: true })
await apiService.post('/clients', payload, { auth: true })
await apiService.put('/clients/42', payload, { auth: true })
await apiService.patch('/clients/42', payload, { auth: true })
await apiService.supprimer('/clients/42', { auth: true })
await apiService.requete({ method: 'HEAD', path: '/sante' })
await apiService.requeteDetaillee({ method: 'GET', path: '/clients', auth: true })

`GET`

Lire une ressource ou une liste.

`POST`

Créer une ressource ou déclencher une action.

`PUT`

Remplacer une ressource complète.

`PATCH`

Modifier partiellement une ressource.

`DELETE`

Supprimer une ressource.

Exemple propre : page → client API → backend

function creerClientApiClients(apiService) {
  return {
    lister() {
      return apiService.get('/clients', { auth: true })
    },
    creer(payload) {
      return apiService.post('/clients', payload, { auth: true })
    },
    modifier(id, payload) {
      return apiService.patch(`/clients/${id}`, payload, { auth: true })
    },
    supprimer(id) {
      return apiService.supprimer(`/clients/${id}`, { auth: true })
    },
  }
}
Règle API

Page = interface. Client API = logique réseau. `apiService` = transport HTTP. Backend = traitement réel.

8. Pages

Pages, routage et navigation

Source de vérité du routage

Le point d’entrée routage est `routage/routes.json`. Il déclare :

  • la page d’entrée
  • les routes
  • les guards éventuels
{
  "pageEntree": "login",
  "routes": {
    "login": "pages/login/login",
    "index": "pages/index/index",
    "inscription": "pages/inscription/inscription"
  }
}

Créer une page

npm run create:page -- profil

Cette commande crée :

  • `pages/profil/profil.js`
  • `pages/profil/profil.json`
  • `pages/profil/profil.wxml`
  • `pages/profil/profil.wxss`
  • la route associée

Synchroniser

npm run sync:pages

Cette commande aligne `routes.json`, `app.json` et `routes.runtime.js`.

Réparer ou supprimer

npm run repair:pages
npm run remove:page -- profil

Naviguer depuis une page

const { navigation } = serviceApp()

await navigation.allerA('index')
await navigation.remplacerPage('login')
Important

Modifier seulement `routes.json` ne crée pas les fichiers d’une page. Le routeur décrit la navigation. `create:page` crée la page physique.

9. Runtime applicatif

État global, lifecycle, bus et storage

Ce qu’il ne faut pas mélanger

`data` / `setData`

État visuel local de la page.

`gestionnaireEtat`

État global de l’application et bus d’événements.

`storage` / `sessionService`

Persistance locale, tokens, restauration et synchronisation.

Helpers de page

const pageHelpers = creerPageHelpers({
  nomPage: 'profil',
  dataInitiale: {
    nomActuel: 'Invite',
  },
})

Page({
  behaviors: [loadingBehavior],
  data: pageHelpers.dataInitiale(),

  onLoad() {
    pageHelpers.onLoad(this, () => {
      pageHelpers.abonner(this, () => {
        const { profilService } = serviceApp()
        return profilService.onNomChange((nom) => {
          this.setData({ nomActuel: nom || 'Invite' })
        })
      })
    })
  },

  async onShow() {
    await pageHelpers.onShow(this, async () => {
      this.rechargerEtat()
    })
  },

  onUnload() {
    pageHelpers.onUnload(this)
  },
})

État global

const { gestionnaireEtat } = serviceApp()

gestionnaireEtat.set('utilisateur.nom', 'Aicha')
const nom = gestionnaireEtat.get('utilisateur.nom', 'Invite')
gestionnaireEtat.remove('utilisateur.nom')

const stop = gestionnaireEtat.onState('utilisateur.nom', (value) => {
  console.log(value)
})

stop()

Bus événementiel

const { gestionnaireEtat } = serviceApp()

const unsubscribe = gestionnaireEtat.on('ui.refresh', (payload) => {
  console.log(payload.source)
})

gestionnaireEtat.emit('ui.refresh', { source: 'login' })
gestionnaireEtat.once('ui.modal.close', () => console.log('une fois'))
gestionnaireEtat.off('ui.refresh')
unsubscribe()

Services métier au-dessus de l’état

const {
  profilService,
  utilisateurService,
  compteService,
  sessionService,
} = serviceApp()

profilService.mettreNom('Aicha')
utilisateurService.hydraterUtilisateur({ id: 'u1', nom: 'Aicha', email: 'aicha@mail.com' })
compteService.hydraterCompte({ id: 'c1', type: 'ADMIN', nom: 'Compte principal' })
await sessionService.sauvegarderToken('abc123')
Règle lifecycle

`onLoad` démarre, `onShow` relit, `onUnload` nettoie. Et si tu veux garder ça propre, passe par `pageHelpers`.

11. Formulaires

Validateurs, messages centralisés et `formHelper`

Le framework centralise maintenant les validateurs dans `utils/validators/` avec messages mutualisés et configuration de locale.

Exemple simple

const { v } = require('../../utils/validators')

const regles = {
  prenom: [v.required('Prénom obligatoire.'), v.prenom()],
  nom: [v.required(), v.nom()],
  email: [v.required(), v.email()],
  telephone: [v.required(), v.telephoneSenegal()],
  motDePasse: [v.required(), v.minLength(8), v.containsUppercase(), v.containsDigit()],
  confirmation: [v.required(), v.sameAs('motDePasse', 'Les mots de passe doivent être identiques.')],
}

Exemples de validateurs déjà fournis

Texte

`required`, `minLength`, `maxLength`, `pattern`, `username`, `nom`, `prenom`, `nomComplet`, `nomSociete`

Contact

`email`, `telephoneSenegal`, `telephoneMobileSenegal`, `telephoneInternational`, `url`, `urlSecure`, `domaine`

Numérique

`numeric`, `integer`, `decimal`, `min`, `max`, `montant`, `pourcentage`, `latitude`, `longitude`

Sécurité

`strongPassword`, `otpCode`, `sameAs`, `containsUppercase`, `containsDigit`

Formats

`uuid`, `ipv4`, `ipv6`, `ip`, `hexColor`, `json`, `mimeType`, `fileExtension`

Exemple avec `formHelper`

const form = creerFormHelper({
  champs: {
    prenom: '',
    email: '',
    telephone: '',
  },
})

Page({
  data: {
    ...form.dataInitiale(),
  },

  handleInputChamp(e) {
    form.handleInput(this, e)
  },

  soumettre() {
    const validation = form.valider(this, regles)

    if (!validation.ok) {
      return
    }

    console.log(validation.valeurs)
  },
})

12. Cas réels

Comment utiliser le framework dans les cas les plus fréquents

1

Brancher un backend Bearer classique

Modifie `config/environments.js` pour le `baseUrl`, puis `config/authProvider.js` pour `transport`, `loginFields` et `endpoints`.

// environments.js
api: Object.freeze({
  baseUrl: 'https://api.exemple.com',
  prefix: '/v1',
  timeoutMs: 10000,
}),
auth: Object.freeze({
  strategy: 'api',
  redirectAfterLogin: 'index',
  redirectIfGuest: 'login',
}),

// authProvider.js
transport: 'bearer',
loginFields: Object.freeze({
  identifiant: 'email',
  motDePasse: 'password',
}),
endpoints: Object.freeze({
  login: '/auth/login',
  me: '/auth/me',
  logout: '/auth/logout',
})
2

Créer un CRUD propre

Ne mets pas les requêtes HTTP directement dans la page. Crée un client API métier.

function creerClientApiClients(apiService) {
  return {
    lister() {
      return apiService.get('/clients', { auth: true })
    },
    creer(payload) {
      return apiService.post('/clients', payload, { auth: true })
    },
  }
}

Page({
  async onShow() {
    const { apiService } = serviceApp()
    const clientApi = creerClientApiClients(apiService)
    const clients = await clientApi.lister()
    this.setData({ clients })
  },
})
3

Faire une page de scan QR

Le bon point d’entrée est toujours `configurationAppareil`.

Page({
  async scannerProduit() {
    const { configurationAppareil } = serviceApp()
    const resultat = await configurationAppareil.scannerCode()

    if (!resultat || !resultat.result) {
      return
    }

    this.setData({
      codeScanne: resultat.result,
    })
  },
})
4

Construire un formulaire d’inscription sans logique répétée

Utilise `creerFormHelper` et `v.*` pour centraliser saisie, validation et messages.

const { creerFormHelper } = require('../../utils/helpers/formHelper')
const { v } = require('../../utils/validators')

const form = creerFormHelper({
  champs: {
    prenom: '',
    email: '',
    telephone: '',
  },
})

const regles = {
  prenom: [v.required(), v.prenom()],
  email: [v.required(), v.email()],
  telephone: [v.required(), v.telephoneSenegal()],
}

14. Règles de projet

FAQ, conventions et anti-patterns

Est-ce que `PMTMiniApp` embarque le backend ?

Non. Il embarque le frontend, la structure, la configuration, les services et la manière propre de consommer un backend.

Où configurer un backend ?

Commence par `config/environments.js`, puis `config/authProvider.js` si le sujet touche l’auth.

Où mettre une donnée ?

UI locale : `data`. Valeur globale partagée : service métier ou `gestionnaireEtat`. Valeur persistante : `sessionService` ou storage.

Puis-je lire l’IMEI ?

Non, pas dans le cadre officiel WeChat Mini Program. Le framework le documente explicitement comme indisponible.

Puis-je publier sans OTP npm ?

Seulement si ton compte ou ton token npm le permet. Sinon, la publication réclame un OTP ou une auth web npm.

À éviter

Écrire directement dans `globalData.noyau.state`, mettre toute la logique réseau dans une page, ou créer des accès natifs hors des services exposés. Le framework t’évite précisément ce type de dérive.
